# ⏰ Deadline Clock

> **Train your process, not your panic.**

Deadline Clock is a workflow-based timer built specifically for **DSA, LeetCode, competitive programming, and coding interview preparation**.

Unlike traditional Pomodoro timers, Deadline Clock doesn't just measure time—it guides you through every stage of solving a coding problem, from understanding the question to analyzing your accepted solution.

It was built around one simple realization:

> **Without a deadline, I wasn't solving problems—I was expanding them to fill all the time I had.**

---

# 🎯 Why I Built Deadline Clock

While practicing DSA, I realized that my biggest problem wasn't solving difficult questions.

It was **how I approached them.**

A typical problem looked like this:

- I'd spend **3–4 hours** on a single LeetCode problem.
- I'd reread the same question multiple times because I never fully understood it the first time.
- My accepted solution usually worked, but it was filled with unnecessary `if`/`else` conditions because I never analyzed my own code.
- As soon as the solution was accepted, I'd move on to the next problem without understanding **why** it worked.
- A few weeks later, many of those same problems felt completely new again.

I also noticed something else.

I always felt **slow**.

Not because I couldn't solve problems.

Because I had **no structure and no deadlines**.

If a problem wasn't solved, I simply kept thinking, rereading, tweaking my approach, and convincing myself that "just five more minutes" would finally solve it.

At the same time, I noticed another problem that many of my friends—and I—also faced.

Sometimes we did the exact opposite.

Instead of spending too long on a problem, we gave up too early.

We looked at hints after only a few minutes because getting stuck felt uncomfortable, even though we probably hadn't spent enough time thinking.

Eventually I realized that DSA students often fall into one of two extremes:

- **We spend too long on one problem.**
- **Or we don't spend enough time thinking before giving up.**

Neither approach helps you become a better problem solver.

While discussing these study habits and experimenting with different ways to improve them, one idea stood out.

> **What if every stage of solving a problem had its own deadline instead of giving the entire problem unlimited time?**

The goal wasn't to make me solve faster.

The goal was to make me spend **the right amount of time on the right activity**.

So I designed a workflow for myself.

Instead of treating every problem as one huge task, I broke it into smaller phases.

- Read the problem carefully once.
- Restate it in my own words.
- Think independently before using hints.
- Compare my solution with the optimal one.
- Analyze my own code instead of immediately moving on.
- Write notes about the pattern instead of just the final answer.

If I couldn't solve the problem within that stage, I stopped extending the timer forever.

Instead, I switched to learning the solution properly, analyzed it, made notes, and returned to it later during revision.

Unexpectedly, this made unfinished problems stay in the back of my mind.

When I came back later, I naturally approached them with more urgency and better focus instead of endlessly grinding in one sitting.

This idea is related to a psychological phenomenon known as the **Zeigarnik Effect**, which suggests that unfinished tasks tend to remain mentally active and are easier to return to later.

After following this workflow consistently, I noticed several improvements.

- I stopped spending entire evenings on one problem.
- I stopped giving up after only a few minutes.
- I reread problem statements much less.
- I analyzed my accepted solutions instead of immediately moving on.
- I remembered patterns instead of memorizing solutions.
- Most importantly, studying finally felt structured.

At first, this workflow was something I built purely for myself.

But after seeing how much it changed my own practice, I realized many students preparing for coding interviews probably struggle with the exact same things.

So I turned that workflow into **Deadline Clock**.

It isn't designed to make you solve more questions.

It's designed to help you learn more from every question you solve.

# 🚀 The Workflow

```text
👀 Read & Restate
        ↓
🧠 Solve (No Hints)
        ↓
💡 Hints Allowed
        ↓
❓ Did you solve it?
      /          \
    Yes          No
     |            |
⚡ Compare     📖 Learn Solution
      \          /
        🔍 Analyze
             ↓
          📝 Notes
```

Every stage has a purpose.

The goal isn't simply to solve more problems.

It's to **learn more from every problem you solve.**

# ☕

## Smart Break System

Breaks are based on **continuous focus time**, not individual workflow stages.

The timer never interrupts your current phase.

Instead:

- Finish the current step.
- If your accumulated focus time exceeds the configured threshold (default **30 minutes**), Deadline Clock recommends taking a short break.
- After the break, focus time resets and the workflow continues.
